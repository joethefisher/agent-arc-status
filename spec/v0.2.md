# Agent Arc Status Protocol: Specification v0.2

**Status:** Draft
**Version:** 0.2.0
**Last updated:** 2026-06-14
**Editor:** Joe Fisher

---

## Abstract

This document specifies the Agent Arc Status Protocol ("the Protocol"), a JSON-based event schema and emission discipline for reporting progress on a unit of authorized agent work, called an **arc**. The Protocol defines five phases (`started`, `milestone`, `heartbeat`, `done`, `blocked`), a canonical event shape, a default cadence rule with a silence backstop, and conformance rules for emitters and consumers. The Protocol is transport-agnostic: events may be delivered via webhooks, message queues, MCP, stdout/stderr, or any other channel that conveys structured data.

The keywords MUST, MUST NOT, SHOULD, SHOULD NOT, MAY, REQUIRED, and OPTIONAL in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) when, and only when, they appear in all capitals.

## 1. Introduction

### 1.1 Motivation

Autonomous agent work increasingly spans minutes to hours. Existing observability standards (e.g. OpenTelemetry) model trace/span lifecycles well, but assume sub-second to single-digit-second durations and offer no first-class notion of *progress within a span*. Agent platforms (e.g. LangChain/LangSmith, LangFuse) provide proprietary observability but no portable vocabulary that works across frameworks. As a result, every team building long-running agents reinvents progress reporting, producing siloed status surfaces that cannot interoperate.

The Protocol addresses one specific gap: **a portable, framework-agnostic event vocabulary for reporting progress on a long-running unit of authorized agent work**, suitable for direct human consumption, dashboards, and consumption by parent or sibling agents.

### 1.2 Scope

The Protocol is in scope for:

- Reporting progress on a unit of authorized agent work that takes longer than a small cadence floor (default: 5 minutes wall-clock).
- Reporting completion or hard blockers on such work.
- Threading multiple events into a single arc timeline via a stable identifier.

The Protocol is out of scope for:

- Full distributed tracing (use OpenTelemetry).
- Per-tool-call or per-LLM-message logging (use platform observability).
- Inter-agent RPC / capability negotiation (use MCP or equivalent).
- Workflow orchestration (use a workflow engine).

### 1.3 Design principles

1. **Minimal vocabulary.** Five phases. No subtype hierarchy. Adding semantics requires a spec revision.
2. **Self-contained events.** Each event MUST be interpretable without recourse to prior events in the arc.
3. **Transport-agnostic.** The Protocol defines the payload and emission rules, not the delivery mechanism.
4. **Easy to emit, easy to consume.** A conformant emitter is implementable in under 100 lines in any modern language. A conformant consumer is implementable in fewer.
5. **Receiver renders.** Emitters MUST NOT pre-format payloads for any specific presentation surface. Consumers render to their context.

## 2. Terminology

- **Arc.** A unit of authorized agent work with a defined beginning and end. An arc has a single owner (the emitter) and zero or more observers (consumers). An arc that does not exceed the cadence floor SHOULD NOT emit events.
- **Phase.** One of the five enumerated values describing the lifecycle position of an arc at the moment of emission.
- **Event.** A single `arc.status` JSON object emitted by an arc owner.
- **Emitter.** The agent or process that owns and emits events for an arc.
- **Consumer.** Any process (agent, dashboard, log sink, human-facing surface) that receives and acts on events.
- **Cadence floor.** The minimum expected arc duration below which the emitter SHOULD NOT emit events. Default: 5 minutes.
- **Silence window.** The maximum interval during which an active arc may emit no events before a `heartbeat` is REQUIRED. Default: 20 minutes.

## 3. The arc.status event

### 3.1 Wire format

Each event is a single JSON object. The canonical event type identifier is `arc.status`. When the event is wrapped by a delivery transport (e.g. a webhook envelope with an event-type header), the transport SHOULD use `arc.status` as the event type.

### 3.2 Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `arc_id` | string (UUID) | yes | A stable identifier for the arc. All events in one arc MUST share an `arc_id`. RFC 4122 UUIDs are RECOMMENDED. |
| `phase` | string (enum) | yes | One of `started`, `milestone`, `heartbeat`, `done`, `blocked`. See §4. |
| `title` | string | yes | A short human-readable label for this event. MUST be ≤ 200 characters and MUST NOT contain newlines (use `body` for multi-line content). |
| `body` | string | no | A longer freeform body. MAY contain markdown. No length cap is defined; consumers MAY truncate. |
| `step` | integer | no | The 1-indexed step number within a sequenced arc. MUST be ≤ `total` if both are present. |
| `total` | integer | no | The total number of steps in a sequenced arc. MUST be ≥ 1 if present. |
| `eta_minutes` | number | no | The emitter's current estimate of minutes remaining to `done`. MUST be ≥ 0 if present. |
| `sent_at` | string (RFC 3339) | yes | The UTC timestamp at which the event was generated. RFC 3339 `date-time`; canonical form `YYYY-MM-DDTHH:MM:SS[.fff]Z`, where `T` and `Z` MUST be uppercase. A numeric offset `±HH:MM` MAY replace `Z`. Fractional seconds are OPTIONAL (1–9 digits). Leap seconds (`:60`) MUST NOT be used, and the value MUST be a valid calendar date. The normative grammar is the `sent_at` `pattern` in [`schema.json`](schema.json); the reference validator applies the identical pattern plus a calendar-validity check. |
| `arc_kind` | string | no | An optional, free-form label naming the kind of work (e.g. `build`, `migration`, `triage`). Reserved for consumer routing; the Protocol assigns it no semantics. |
| `protocol_version` | string | no | The Protocol version the emitter targets, e.g. `0.2`. Consumers comparing it SHOULD compare `major.minor` only (§10). |

Fields not listed above are RESERVED. Emitters MAY include additional fields prefixed with `x_` (e.g. `x_my_app_correlation_id`) for application-specific extensions; the schema enforces this (`additionalProperties: false` plus a `^x_` pattern), so a non-`x_` unknown field is invalid at the emitter. Consumers, for forward compatibility, MUST still ignore `x_` fields they do not understand (§7.1).

### 3.3 JSON Schema

The normative JSON Schema for an `arc.status` event is provided at [`spec/schema.json`](schema.json). Implementations MAY use it for validation; behavior described in this document is normative where the schema is silent.

### 3.4 Minimal valid event

```json
{
  "arc_id": "00000000-0000-4000-8000-000000000001",
  "phase": "started",
  "title": "build Pulsefeed v0.1",
  "sent_at": "2026-06-14T02:00:00.000Z"
}
```

## 4. Phases

The Protocol defines exactly five phases. Adding a phase requires a spec revision.

### 4.1 `started`

Marks the beginning of an arc. The emitter MUST send exactly one `started` event per arc, and it MUST be the first event emitted for that arc. The `title` SHOULD describe the arc's objective.

An arc whose expected wall-clock duration is below the cadence floor (§5.1) SHOULD NOT emit a `started` event (or any other event).

### 4.2 `milestone`

Marks a meaningful checkpoint within an active arc. Examples: a major sub-step completed, a service brought up, a test suite passed, a commit landed, an external action succeeded. The `title` SHOULD describe what just happened or what is now beginning. The `step` and `total` fields MAY be present if the arc is sequenced.

An emitter MAY emit zero or more `milestone` events per arc. Emitters SHOULD NOT emit `milestone` events for trivial sub-steps (e.g. individual tool calls); see §5.3 for anti-patterns.

### 4.3 `heartbeat`

Indicates that an arc is still active when the silence window (§5.2) would otherwise elapse with no other event. The `title` SHOULD describe the current activity (e.g. `"still working: debugging systemd unit"`).

`heartbeat` events MUST NOT carry semantic progress information that would be conveyed better by `milestone`; if the emitter has real news, the emitter SHOULD emit `milestone` instead and reset the silence timer.

An emitter MAY emit zero or more `heartbeat` events per arc.

### 4.4 `done`

Marks successful completion of an arc. The emitter MUST send exactly one `done` event per arc, and it MUST be the last event emitted for that arc. The `title` SHOULD summarize the final deliverable.

An emitter MUST verify completion from the consumer's vantage point before emitting `done` (i.e. the deliverable is visible on the surface the consumer expects). Emitting `done` for incomplete work is a conformance violation.

### 4.5 `blocked`

Indicates that the arc cannot make progress without input from outside the emitter's authority (typically a human gate, a permission decision, or external state that is not arriving). The `title` SHOULD state the blocker concisely; the `body` SHOULD describe what would unblock it.

Two `blocked` events MUST be separated by at least one non-`blocked` event: when the arc resumes, the emitter MUST emit a `milestone` describing the unblock before emitting any further `blocked`. (v0.1 scoped this "for a given blocker", but the wire format carries no blocker identity, so the rule is stated structurally and is enforced by the reference sequence validator.) An arc MAY conclude with `blocked` rather than `done` if the blocker is terminal; see §4.6. A `blocked` arc is not silently working, so the silence backstop (§5.2) does not apply while it is blocked; `heartbeat` is not a legal event immediately after `blocked`.

### 4.6 Phase ordering

The legal phase sequence for an arc is:

```
started → (milestone | heartbeat | blocked)* → (done | blocked-as-terminal)
```

Where:

- An arc MUST begin with `started` and end with either `done` or a terminal `blocked`.
- `milestone`, `heartbeat`, and `blocked` events may interleave in any order between `started` and the terminal event.
- After a `blocked`, the next event in the arc MUST be either `milestone` (resumed), `done` (resolved and completed), or no further events (terminal `blocked`).
- An arc that emits `done` MUST NOT emit further events; consumers MAY drop subsequent events for that `arc_id` as protocol violations.

The reference sequence validator (`validateSequence`) enforces this grammar for a **complete** arc, including the terminal-event requirement and the no-consecutive-`blocked` rule (§4.5). To validate an in-flight prefix that has not yet reached its terminal event, callers pass `{ partial: true }`, which relaxes only the terminal-last check.

## 5. Cadence

### 5.1 The cadence floor

The Protocol defines a default cadence floor of **5 minutes** of expected wall-clock duration. An emitter SHOULD NOT emit events for arcs whose expected duration is below the floor. The rationale: events on short arcs are noise. The outcome itself arrives before status would have been useful.

Implementations MAY parameterize the cadence floor. The default is RECOMMENDED for general-purpose use; reduce it only when consumers are known to benefit from finer-grained progress (e.g. customer-facing progress bars for sub-minute operations).

The floor is a **delay threshold, not a prediction**. Expected duration is unknowable at `started` time, so an emitter MUST NOT permanently suppress an arc on the basis of a guess: an arc that has not emitted `started` but whose actual elapsed time crosses the floor MUST emit `started` at that point (retroactively) and thereafter participate in the silence backstop. This keeps an arc that was expected to be short but runs long from staying invisible, which is the protocol's worst failure mode. The reference `CadenceController` implements this.

### 5.2 The silence window

The Protocol defines a default silence window of **20 minutes**. When an active arc has emitted no event for the duration of the silence window, the emitter MUST emit a `heartbeat`. The silence window resets on any event.

The rationale: silence is ambiguous. A 20-minute gap may mean "still working" or "crashed." The silence backstop narrows that ambiguity: if even the heartbeat is missing, the arc has likely failed and the consumer can take action.

Because the heartbeat is self-emitted, it cannot fire in the one case the backstop most needs to surface: when the emitter's own work loop has stalled. Emitters MUST therefore drive the backstop from a timer or supervisor **independent of the work loop**, such as an in-process timer the loop cannot starve, or an external watchdog (see [adoption guide §3](../docs/adoption.md)). Correspondingly, absence of a heartbeat is **necessary but not sufficient** evidence of failure: it may instead mean a clean exit whose `done` was lost, or a dropped delivery. Consumers therefore treat it as a stall *signal* (§7.4), not proof.

Implementations MAY parameterize the silence window. The default is RECOMMENDED.

### 5.3 Anti-patterns

The following are explicit non-conformances or strong disrecommendations:

- **Per-tool-call milestones.** Emitting a `milestone` for every internal tool call or LLM round trip is noise. Reserve `milestone` for events the consumer would describe as progress.
- **Heartbeat spam.** Emitting `heartbeat` more often than every silence-window interval defeats its purpose.
- **Empty `milestone` payloads.** A `milestone` whose `title` says only "still working" SHOULD be a `heartbeat` instead.
- **Premature `done`.** Emitting `done` before the deliverable is visible to the consumer is a conformance violation, not a stylistic choice.
- **Multiple `started` or `done`.** Each MUST occur exactly once per arc.

The heuristic for emitting: *if the consumer asked right now "how is it going?", what is the answer?* Emit that. Then keep working.

## 6. Emitter conformance

Emitter conformance has two tiers. The distinction matters because a conformance rule a third party cannot check is a wish, not a contract.

**Machine-verifiable.** Checkable from a captured event stream, and enforced by the reference implementation:

1. **Schema validity.** Every emitted event conforms to the JSON Schema (§3.3).
2. **Phase legality.** The sequence of phases per `arc_id` conforms to §4.5–§4.6.

**Operational.** Cannot be proven from the event payloads alone; a conformance harness needs the additional inputs named here:

3. **Cadence backstop.** The emitter never allows an active arc to exceed the configured silence window with no event. *Verifying this requires per-event receipt timestamps and the configured silence window; it cannot be judged from payloads alone.*
4. **Unique `arc_id`.** Each arc has a globally-unique `arc_id`; reuse across arcs is a violation. *Verifying this requires observing multiple arcs: a harness watching N arcs MUST see N distinct `arc_id`s; a single-arc validator cannot detect reuse.*
5. **Honest `done`.** The emitter does not emit `done` until the deliverable is verified from the consumer's vantage point. *This is an operational discipline with no payload-observable proxy; it cannot be validated mechanically.*

The planned conformance suite (§12) targets the machine-verifiable tier, plus the cadence backstop given a timing harness. Conformant emitters MAY support additional features (`x_`-namespaced extensions, custom transports, per-consumer formatting) provided they do not violate the rules above.

## 7. Consumer conformance

A consumer is **conformant** if it satisfies all of the following:

1. **Schema tolerance.** The consumer validates events against the JSON Schema and ignores unknown fields (forward compatibility).
2. **`arc_id` threading.** The consumer groups events by `arc_id` when presenting timelines.
3. **Phase respect.** The consumer treats `done` as terminal and SHOULD NOT display further events for that `arc_id` as progress.
4. **Silence detection.** The consumer treats absence of any event (including `heartbeat`) beyond the silence window as a stall signal, measured against the consumer's **local receipt time** of the arc's most recent event, never against `sent_at`, which is subject to clock skew and delivery delay (§7.5). A stall is a signal, not proof of failure (§5.2); consumers MAY surface it to humans, attempt recovery, or take whatever action suits their context.
5. **No ordering assumption.** The consumer does not assume events arrive in order. Out-of-order delivery is possible on lossy transports; consumers SHOULD reorder by `sent_at` for display and threading. `sent_at` is for ordering and display only and carries **no monotonicity guarantee** across a real transport; consumers MUST NOT use it as a liveness clock (see §7.4).

## 8. Transport

The Protocol is transport-agnostic. The following bindings are common and RECOMMENDED.

### 8.1 Webhook binding

Events are delivered as the body of an HTTP `POST`. The HTTP request SHOULD carry:

- `Content-Type: application/json`
- `X-Webhook-Event-Type: arc.status` (or transport-defined equivalent)
- A delivery-unique identifier header (e.g. `X-Webhook-Delivery-Id`)
- A timestamp header for replay protection (e.g. `X-Webhook-Timestamp`)
- An HMAC signature header for authenticity (e.g. `X-Webhook-Signature`)

See §9 for security recommendations.

### 8.2 Message-queue binding

Events are published to a queue or topic. The queue message body MUST be the event JSON. The queue message metadata SHOULD carry the event type and `arc_id` for filtering.

### 8.3 Stdout binding

Events are written to standard output as JSON Lines. Each line is one event. This binding is useful for local development, log shipping, and tail-based consumers.

### 8.4 Other transports

The Protocol is compatible with any transport that conveys structured payloads. MCP tool results, SSE streams, gRPC messages, and Kafka topics are all valid bindings provided the event JSON is preserved.

## 9. Security considerations

### 9.1 Authentication

The Protocol does not specify authentication. Implementations SHOULD use transport-level authentication appropriate to the transport (e.g. HMAC signing for webhooks, IAM policies for cloud queues, mTLS for service-to-service).

### 9.2 Replay protection

Events are point-in-time progress reports. They are not commands. Replay of a past event SHOULD be tolerated by consumers (it is informational), but consumers MAY reject replays older than a transport-defined window to limit the damage of leaked credentials.

### 9.3 PII and sensitive content

Emitters SHOULD NOT include credentials, personal data, or confidential business information in `title` or `body` unless the entire delivery channel is trusted. Consumers MAY persist events indefinitely; assume durability when deciding what to include.

### 9.4 Trust boundary

Events are **untrusted data** to their consumers. A consumer that takes action based on event contents (e.g. an agent that parses a `title` and acts on it) MUST treat the content as input, not instruction. The Protocol exists to report progress, not to authorize side effects.

## 10. Versioning

The Protocol uses semantic versioning. The version is reported in the OPTIONAL `protocol_version` field on events and SHOULD also appear in transport metadata. Consumers comparing `protocol_version` SHOULD compare only `major.minor` and ignore any patch component. `0.1` and `0.1.0` are equivalent, and a consumer that understands `0.2` accepts any `0.2.x`.

- **Patch (0.1.0 → 0.1.1):** Editorial fixes only. Wire-compatible.
- **Minor (0.1 → 0.2):** Additive, backward-compatible changes. New OPTIONAL fields, new RECOMMENDED behaviors, clarifications. Existing conformant emitters and consumers remain conformant.
- **Major (0.x → 1.0, 1.x → 2.0):** Breaking changes. New REQUIRED fields, removed phases, semantic shifts. Major version bumps SHOULD be rare.

The current draft is **v0.2**. The Protocol will reach v1.0 after at least one external implementation reports successful interoperability.

## 11. Compatibility with other systems

### 11.1 OpenTelemetry

The Protocol is complementary to OpenTelemetry, not a replacement. An arc MAY be a logical parent of one or more OTel traces; the `arc_id` MAY be carried as an OTel attribute on spans for correlation. The Protocol's phases correspond loosely to OTel span status (`started` ≈ span start, `done` ≈ span end with `OK`, terminal `blocked` ≈ span end with `ERROR`), but the Protocol's emission discipline (cadence floor, silence backstop, milestone semantics) has no OTel equivalent.

### 11.2 Agent framework events

Most agent frameworks (LangChain, AutoGen, CrewAI) emit internal events for tool calls, LLM round trips, and intra-graph transitions. These are at a finer granularity than `arc.status` and should NOT be translated 1:1. A framework integration SHOULD aggregate framework-internal events into `milestone` events at human-meaningful boundaries.

### 11.3 Workflow engines

Workflow engines (Inngest, Temporal, Airflow) have their own progress models. The Protocol can coexist: a workflow step MAY emit `arc.status` events to a separate consumer for human-facing visibility while the workflow engine handles retry and orchestration internally.

### 11.4 A2A (Agent2Agent)

[A2A](https://github.com/a2aproject/A2A) is a protocol for agents to discover one another and delegate tasks across vendors. It standardizes *how* one agent hands a task to another and exchanges messages about it; it does not standardize the in-flight liveness/progress signal a long-running delegated task emits while it executes. The two compose: an A2A Task MAY carry an `arc_id`, and the delegate emits `arc.status` events the delegator consumes to know the delegated work is alive and moving. Reach for A2A to delegate or discover an agent; reach for this Protocol to report or observe progress on the delegated work.

## 12. Open questions

The following are explicitly under-specified in v0.2 and are candidates for a future revision.

1. **Sub-arc / parent-arc relationships.** When Agent A delegates work to Agent B, B's arc is logically nested under A's. v0.2 keeps `arc_id` flat. Multi-agent delegation is expected to introduce `parent_arc_id` (and possibly a `root_arc_id` for tree grouping) in a future minor. Until then, emitters that need parent/child threading SHOULD use `x_parent_arc_id`; the final field shape will be derived from real delegation usage, and the `x_`-prefixed convention promoted to a first-class field once that shape is confirmed.
2. **Standard `arc_kind` taxonomy.** v0.2 leaves `arc_kind` free-form for consumer routing. A future revision may define common kinds (`build`, `migration`, `analysis`, `outbound`, etc.).
3. **Cancellation.** v0.2 does not define a `canceled` phase. Cancellation is currently expressed as terminal `blocked` with a `body` describing the reason. Whether to add a first-class `canceled` phase is open, pending an adopter signal that "couldn't continue" and "chose not to continue" need to be distinguished on the wire.
4. **Progress percentage.** v0.2 uses `step`/`total` for discrete progress. Whether to add a continuous `progress_pct` for non-discretized work is open.
5. **Conformance test suite.** v0.2 does not yet ship a full conformance test suite, though the reference implementation now exercises the machine-verifiable tier (§6) and the cadence module. A complete suite is planned once a second independent implementation exists.

Feedback on any of the above via the [issue tracker](https://github.com/joethefisher/agent-arc-status/issues) or as PRs against [`spec/`](.) is welcome. See [CONTRIBUTING.md](../CONTRIBUTING.md) for the process.
